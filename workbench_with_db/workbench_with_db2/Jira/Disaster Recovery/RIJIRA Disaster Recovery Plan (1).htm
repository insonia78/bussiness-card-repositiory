Introduction:

Tool Description:

JIRA, built by Atlassian, is an issue tracking tool that allows a company to track issues, assign work, and complete tasks within the company.  It uses a graphical user interface and can be used on Windows, Linux, or Mac operation systems.

JIRA's allows the client to create projects, issues, custom workflows to meet their business needs.  A user is able to create a project, and within this project, issues can be created.  Issues can follow a certain workflow and can be assigned to a user or group.  In addition, JIRA is able to keep track of users, and assign permissions to each user to control what is available to each user.  

By putting all the company's issues into one place, it allows a company to be much more organized, and allow issues within the company to be fixed faster.  

Why GTECH Uses JIRA?

Atlassian's JIRA is used by GTECH as a defect tracking issue tool.  It allows GTECH to post issues about its products and services, and assign these issues to be fixed by other users.  JIRA is used by most employees, especially GTECH's software developers, and is currently installed on a server named RIJIRA.  

Current Version:

GTECH currently uses JIRA 5.2.11.

Backup Procedures:

Database Backup:

·         An xml backup file gets created every 12 hours on RIJira.  This backups both the application and the database within Jira.  However, this does not back up the attachments.

·         Every night around 9:00pm E.T, the latest xml file gets automatically copied over to what is now RIJira01.  A little while after that all of the attachments on RIJira are copied over to RIJira01.

 

 

 

 

Restore Procedures:

 

*Important: Currently, the only servers for JIRA are located at the Data Center in Rhode Island.  There is talk about creating additional backup servers in a different location in case the Data Center in Rhode Island becomes unavailable. 

 

Database Recovery:

 

If the data becomes corrupted, follow these steps to restore the data.

        Retrieve the most recent backup from the G:\Home\Jira52\export folder.

        Log into Jira 5.2 as a System Administrator.

        In the Jira dashboard, click on the Import & Export link in system tools, and click on Restore System.

        Enter the file name in the File Name text area, and click Restore.

Application Recovery:

 

If only the Jira application is corrupted, and not the data, follow these steps to restore the Jira application.  These steps will re-install the current 5.2 Jira system.

*Note:  <DTOM> = D:\Program File\Apache\Tomcat 6.0\          

<GBAC> = G:\Backups\2013-08-28\  

        Stop the service Tomcat

        Delete (or move) the current Jira folder out of <DTOM>webapps and also erase the Jira directory under <DTOM>work\Catalina\localhost.

        Reset logs in Tomcat

        Replace <DTOM>lib with G:\Backups\6-19-13\tomcat jars

        In the WAR file in webapps\jira\WEB-INF\lib delete slf4j-log4j12-1.6.4 and slf4j-api-1.6.4 (Already in <DTOM>lib)

        [Recompile,] rename and copy the 5.2 WAR file from G:\Downloads\Working WAR Files\atlassian-jira-5.2.11-war\dist-tomcat\tomcat6\atlassian-jira-5.2.11.war 
to G:\Atlassian\JIRA\deploy-war\atlassian-jira.war.

        Start Tomcat, check logs, etc. for any errors.

 

 

 

 

 

 

Server Recovery:

Important: Currently, the only servers for JIRA are located at the Data Center in Rhode Island.  There is talk about creating additional backup servers in a different location in case the Data Center in Rhode Island becomes unavailable.  

Every night around 9:00pm E.T. the latest xml file gets automatically copied over to what is now RIJira01.  A little while after that all of the attachments on RIJira are copied over to RIJira01.

If the RIJIRA suddenly crashed, Platforms could direct traffic and change users permissions to have users use RIJIRA01 until RIJIRA was fixed.

To have RIJIRA01 be used, a request would have to be sent to Platforms through the help desk.

GTECH Jira User Notes for Customer Projects.

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

USER   CONFIGURATION

·        

There will be a project on the customer Jira server corresponding with the one on the GTECH Jira server.    The project key will use the GTECH key preceded by PJ (examples: PJXXX, PJLUNA, PJESOA where the production projects are XXX, LUNA, and ESOA).

 

o   When Jira Administrators create the customer Jira project, they will set the Project Lead to be a designated Customer user.   That Project Lead will be assigned to the Tester role in the customer’s version of the project.

 

o   Jira Administrators will be responsible for adding customer users to the Jira system on the customer server.   The user names need and passwords need to be provided.   Please note that their password must be 8 characters long and have both numeric and alphabetic characters.  The GTECH Project Leads should forward these requests to SRPT_Requests@gtech.com.  

 

o   Please be VERY CAREFUL to select only known names for your site as this is how we restrict project visibility – all customer users will be assigned the Tester role in the project in order for the user to browse the project and assign to other customer users.   It is crucial that no external user is assigned a role in a project other than the one associated with their site – users with any role can browse information in a project.

 

 

 

 

 

PROJECT   CONFIGURATION

·         From within the GTECH project, all issues logged by the customer from the external server will have a single generic lottery Reporter.  The Jira Administrator will create that generic user in the GTECH Jira project and assign it a Tester role.  

 

o   Format of user:   Project Name Lottery (GTECH Project Lead username)

o   Example1:    Sample One Lottery (Sharon Lindeblad)

o   Example2:  RFSS-Spain ONCE Lottery (Viktor Farcic)

 

 

ISSUE   CONFIGURATION

·         Issues created from customer side are limited to: CAT, TIR-Live, and Enhancement Request issue types only.   There will be a corresponding issue in the GTECH Jira system.  The two issues are not numbered the same but are linked programmatically (not using Jira links).

 

·         For each issue submitted by the customer, there is a tab with a link to the customer version of the issue within the GTECH Jira issue for use by GTECH personnel.   The customer does not have a similar link and has no access to the GTECH version of the issue.

 

 

 

·         The workflow for the customer version of the issues will not include all statuses that GTECH versions of the issues go through as directed by GTECH Info Security.   The customer will see the issue as In Progress while it goes through the development and QA process on the GTECH side.

 

CUSTOMER ACTIONS

·         Customers can edit and add attachments before the issue is submitted to GTECH.

 

·         Customers can also Close the issue as Not A Problem prior to submission.

 

·         Any attachments linked to an issue when the customer submits the issue to GTECH will be copied to the GTECH version of the issue.

 

·         The customer will have a Submit Comment to GTECH action available for each issue.   The resulting comments will be displayed in both the customer issue and the GTECH version of the issue with submitter information.  

 

·         Customer users can also assign issues in their system to other customer users.

 

GTECH USER ACTIONS

·         There is a separate action to log comments that GTECH users would like visible to the customer users,  Submit Comment to Customer.   It is available to the current assignee of the issue in all non-closed statuses.

 

o  

If there is no customer version of an issue (it was created within the GTECH Jira system) the Submit Comment to Customer action will not be available/displayed.

 


o   These comments will be propagated to the customer version of the issue but will not include names of GTECH personnel, as directed by GTECH InfoSecurity

o   For internal-only comments, GTECH personnel should use the standard Jira Comment action.

 

·         If GTECH users have an attachment that they want visible to the customer they need to log into the project on the customer Jira server and add the attachment.

 

·         Pass QA, Ready for Production actions cannot be initiated on the GTECH server for those TIR-Live or Enhancement Request issues that have been submitted by the customer.    Also, CAT cannot be by-passed for those issues.

 

o   For issues that are logged/created on the GTECH server the Pass QA, Ready for Production actions are available to the GTECH user

 

·         Pass/Fail CAT actions cannot be initiated on the GTECH server for those issues that have been submitted by the customer.    This is designed such that if an issue is created by the customer, they will be the one to mark it Pass or Fail.   Alternatively, the GTECH Project Lead can log into the customer server and initiate the action from within the customer version of the issue if need be since they have Manager role in that project.

 

o   For issues that are logged/created on the GTECH server the Pass/Fail CAT actions are appropriately available for initiation on the GTECH server

 

·         When an issue is cloned the Remote Jira Link field on the CustomerAccessRef tab must be cleared in the clone.  If not, the workflow will error because it’s trying to reconcile with associated issue in PartnerJira.

 

·         When GTECH users update an Enhancement issue that was created/reported by the customer, the following field values will be propagated to the customer.  Please note this will only occur if Edit function is used to update.   It will not propagate to PartnerJira using Bulk Edit or Quick Edit (version 5).

 

o   Summary

o   Priority

o   Due Date

o   Component/Sched Batch

o   Product (parent and child, if applicable)

o   Site-Specific Product

o   Revenue-Generating

o   Fix Version
NOT  Description, Requester, or Requesting Agency

·         When GTECH users update a CAT or LIVE defect issue that was created/reported by the customer, the following field values will be propagated to the customer.  Please note this will only occur if Edit function is used to update.   It will not propagate to PartnerJira using Bulk Edit or Quick Edit (version 5).

 

o   Summary

o   Priority

o   Severity

o   Due Date

o   Component/Sched Batch

o   Product (parent and child, if applicable)

o   Site-Specific Product

o   Found-In Batch

o   Reproducible

o   Defect Symptom

o   Fix Version

o   Test Description (added Feb 2014)
* NOT Description

GTECH MANAGER ACTIONS

·         GTECH managers will be responsible for adding Batches in the customer project on the customer Jira server to correspond with those on the GTECH Jira server.  You are not required to enter batch details (SAP ID, Start Date, Release Date), however.   Only the exact same name as is in GTECH project will be required.

 

·         GTECH managers will not need to add Versions in the customer project on the customer Jira server to correspond with those on the GTECH Jira server.    A program copying all versions entered in the GTECH Jira project is run every 30 minutes to synchronize versions so that read-only information is available to the customer in the Fix Version field of issues.
